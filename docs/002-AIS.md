## 第一章：Dictionary（字典）

> 所有 AI 必须遵守的统一语言。

### 1.1 Capability Dictionary

#### 定义

**Capability** = AI 系统或产品具备的某种具体能力或功能。

- 客观的：可以通过测试或文档确认
- 可比较的：不同模型可以用同一套 Capability 对比
- **核心资产**：不是 Event，不是 Article

#### 命名规范

```
{domain}.{category}.{capability}
```

全小写，连字符分隔，三级结构，不能多也不能少。

#### 条目格式

```yaml
code: memory.long-term.user
name_en: User Memory
name_zh: 用户记忆
domain: memory
category: long-term
definition: >
  模型能够跨会话记住用户的信息和偏好，
  并在后续交互中使用这些信息。
aliases:
  - personalized-memory
  - cross-session-memory
examples:
  - "Claude remembers your coding style across conversations"
related:
  - memory.session.context
  - memory.project.workspace
version: 1.0
added_at: "2026-07-06"
```

#### 核心 Domain

| Domain | 中文 | 说明 |
|--------|------|------|
| `reasoning` | 推理 | 逻辑、数学、代码、规划 |
| `interaction` | 交互 | 用户与 AI 的交互方式 |
| `tool-use` | 工具使用 | AI 调用外部工具 |
| `memory` | 记忆 | 上下文、长期记忆、项目记忆 |
| `multimodal` | 多模态 | 图像、音频、视频理解/生成 |
| `agent` | 智能体 | 自主 Agent、多 Agent、规划 |
| `integration` | 集成 | API、SDK、插件、嵌入 |
| `deployment` | 部署 | 云端、本地、边缘、设备端 |
| `safety` | 安全 | 护栏、对齐、鲁棒性 |
| `performance` | 性能 | 速度、效率、扩展性 |
| `developer` | 开发者 | 提示工程、微调、评估 |

完整 Capability 树参考 [005-AKG.md](005-AKG.md)。

### 1.2 Entity Dictionary

**Entity** = AI 领域中具有明确身份的对象。

#### 实体类型

### 1.3 Event Dictionary

#### 定义

**Event** = 由多个 Fact 组成的能力变化事件。

#### 事件类型

| Type | 中文 | 定义 |
|------|------|------|
| `model-release` | 模型发布 | 新模型首次发布 |
| `model-update` | 模型更新 | 现有模型重大升级 |
| `feature-added` | 功能新增 | 新增能力 |
| `feature-removed` | 功能移除 | 移除能力 |
| `feature-improved` | 功能改进 | 能力性能提升 |
| `feature-deprecated` | 功能废弃 | 功能即将下线 |
| `api-added` | API 新增 | 新增 API 端点 |
| `api-changed` | API 变更 | API 参数变化 |
| `api-deprecated` | API 废弃 | API 即将下线 |
| `agent-capability` | Agent 能力 | Agent 能力变化 |
| `availability-expanded` | 可用性扩展 | 开放新地区/用户群 |
| `availability-restricted` | 可用性限制 | 限制使用范围 |
| `pricing-changed` | 价格变化 | 定价调整 |
| `limitation-changed` | 限制变化 | 限额调整 |
| `benchmark-updated` | 基准更新 | 评测结果更新 |
| `bug-fixed` | Bug 修复 | 修复问题 |
| `incident` | 故障事件 | 服务中断 |
| `security-update` | 安全更新 | 安全策略更新 |
| `research` | 研究发布 | 论文发布 |
| `partnership` | 合作 | 战略合作（仅影响能力时） |

### 1.5 Fact Dictionary

#### 定义

**Fact** = 从 Signal 中提取的单个事实。

一条新闻可能包含多个 Fact。Fact 是构成 Event 的原子单位。

#### Fact 类型

| Type | 中文 | 示例 |
|------|------|------|
| `model-released` | 模型发布 | "Claude 4.1 Released" |
| `capability-added` | 能力新增 | "Support Memory" |
| `capability-removed` | 能力移除 | "Removed Web Search" |
| `api-available` | API 可用 | "API Available" |
| `price-changed` | 价格变化 | "Price decreased 50%" |
| `availability-changed` | 可用性变化 | "Available in EU" |
| `affected-users` | 影响用户 | "Pro Only" |
| `performance-improved` | 性能提升 | "Speed increased 2x" |

#### 条目格式

```yaml
fact_id: fact-xxx
type: capability-added
subject: model-claude-4-1
capability: memory.long-term.user
value: "cross-session memory up to 100 conversations"
evidence:
  - source: source-anthropic-blog
    url: "https://anthropic.com/news/claude-4-1"
    extracted_from: "Claude 4.1 now supports long-term memory"
confidence: 0.95
detected_at: "2026-07-06T10:00:00Z"
```

#### Fact 的重要性

为什么需要 Fact 层？

示例：
```
新闻：Claude 4.1 今天发布，支持 Memory，开放 API，仅限 Pro 用户。

拆成 Fact：
  1. model-released: Claude 4.1
  2. capability-added: Memory
  3. api-available: Claude 4.1 API
  4. affected-users: Pro Only
```

好处：
- 同一事件的不同方面可以独立分析
- Knowledge Graph 不会因为一条新闻而混乱
- 更容易合并来自不同来源的信息

### 1.6 Evolution Dictionary

#### 定义

**Evolution** = 某个 Capability 或 Entity 的演化历史。

这是本规范最重要的新增内容。我们记录的不是静态快照，而是动态演化。

#### Evolution 类型

| Type | 中文 | 说明 |
|------|------|------|
| `capability-first-seen` | 能力首次出现 | 哪个模型/产品第一个支持 |
| `capability-adopted` | 能力被采用 | 后续模型/产品跟进支持 |
| `capability-removed` | 能力被移除 | 某个模型/产品不再支持 |
| `capability-improved` | 能力改进 | 能力性能提升 |
| `model-evolution` | 模型演进 | 模型版本迭代 |
| `product-evolution` | 产品演进 | 产品功能迭代 |

#### 条目格式

```yaml
evolution_id: evolution-memory-adoption
capability: memory.long-term.user
timeline:
  - date: "2024-02-15"
    entity: model-gpt-4
    event: capability-first-seen
    description: "OpenAI first introduces long-term memory"
  - date: "2024-08-20"
    entity: model-claude-3-5-sonnet
    event: capability-adopted
    description: "Claude adds long-term memory support"
  - date: "2025-01-10"
    entity: model-gemini-1-5-pro
    event: capability-adopted
    description: "Gemini adds memory feature"
  - date: "2025-05-05"
    entity: model-qwen-2
    event: capability-adopted
    description: "Qwen 2 supports long-term memory"
current_status: mainstream
adoption_count: 4
first_seen_at: "2024-02-15"
latest_adoption_at: "2025-05-05"
```

#### Evolution 的价值

```
问题：哪些模型支持 Memory？

答案：从 Evolution 表直接查询 → 4个模型

问题：Memory 从出现到成为主流花了多长时间？

答案：2024-02 → 2025-05 = 15个月

问题：过去一年有哪些能力首次出现？

答案：查询 capability-first-seen 事件 → 列出所有新能力
```

## 第二章：Ontology（本体）

> 定义实体之间的关系。

### 核心关系

| Relation | 中文 | 定义 | 示例 |
|----------|--------|------|------|
| `belongs_to` | 属于 | 子实体属于父实体 | GPT-4 belongs_to OpenAI |
| `powers` | 驱动 | 模型驱动产品 | GPT-4 powers ChatGPT |
| `supports` | 支持 | 模型/产品支持能力 | Claude supports Memory |
| `requires` | 依赖 | 能力依赖另一种能力 | MCP Client requires Tool Use |
| `competes_with` | 竞争 | 实体竞争关系 | Claude competes_with GPT-4 |
| `implements` | 实现 | 产品实现协议 | Claude Desktop implements MCP |
| `evaluated_by` | 被评测 | 模型被基准评测 | GPT-4 evaluated_by MMLU |
| `predecessor_of` | 前身 | 版本演进 | GPT-4 predecessor_of GPT-4o |
| `deprecated_by` | 被替代 | 被新实体替代 | GPT-3.5 deprecated_by GPT-4o |
| `evolved_from` | 演化自 | 能力演化关系 | Memory evolved_from Session Context |

### 关系规则

1. **一个 Model 只能属于一个 Company**（开源模型例外）
2. **一个 Product 可以基于多个 Model**
3. **Capability 支持关系是动态的**（随时间变化）
4. **Event 作用于具体实体**（必须明确作用于哪个实体）
5. **Evolution 记录变化历史**（每个变化都有时间戳）

## 第四章：Scoring（评分）

### Signal Score

```
Signal Score = Source × 0.30 + Content × 0.25 + Recency × 0.15 + Entity × 0.15 + Novelty × 0.15
```

处理规则：
- ≥ 80：P0，立即处理
- 60-79：P1，高优先级
- 40-59：P2，普通
- 20-39：P3，低优先级
- < 20：Discard，丢弃

### Event Score

```
Event Score = Capability Impact × 0.35 + Novelty × 0.20 + Actionability × 0.20 + Source Confidence × 0.15 + User Impact × 0.10
```

### Confidence Score

```
Confidence = min(100, Σ(source_credibility × ai_confidence × cross_validation_bonus))
```

交叉验证 Bonus：
- 1个来源：1.0
- 2个来源：1.2
- ≥3个来源：1.4

### User Value Score

```
User Value = Actionability × 0.40 + Event Score × 0.30 + User Impact × 0.30
```

## 第五章：Naming Convention（命名规范）

### Capability

```
{domain}.{category}.{capability}
```

示例：`memory.long-term.user`

规则：全小写、连字符分隔、三级结构

### Event Type

```
{action}-{target}
```

示例：`feature-added`

规则：全小写、连字符分隔、动词在前

### Entity ID

```
{type}-{name}
```

示例：`model-claude-3-5-sonnet`

规则：全小写、连字符分隔、包含版本号

### 数据库字段

规则：蛇形命名、无缩写、`_at` 后缀表示时间点、`is_` 前缀表示布尔值

示例：`event_score`、`created_at`、`is_official`

### API 端点

规则：RESTful、复数形式、层级用路径、过滤用 query

示例：`GET /events`、`GET /models/{id}/timeline`

## 第六章：Evolution Rules（演进规则）

### 新增 Capability

流程：提案 → Review → 批准 → 回刷

禁止：
- 不允许创建与现有 Capability 含义重叠的新标签
- 不允许因为"叫法不同"就创建新标签（用 aliases 解决）
- 不允许跳过 Review 直接添加

### 新增 Event Type

流程：观察（≥5次同类事件）→ 提案 → Review → 批准

### 新增 Entity

流程：自动检测 → 人工确认 → 添加到 Dictionary → 建立关系

### 修改现有定义

流程：标记 Deprecation → 创建新定义 → 迁移期（3个月）→ 回刷

### 版本管理

```
{major}.{minor}.{patch}
```

- 新增 Capability/Event Type/Entity：minor +1
- 修改定义（不破坏兼容）：patch +1
- 重构结构（破坏兼容）：major +1
